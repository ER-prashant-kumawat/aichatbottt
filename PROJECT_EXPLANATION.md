## 💾 Database Schema

### 1. Users Table
```typescript
{
  _id: Id<"users">           // Auto-generated by Convex
  email: string
  name: string
  createdAt: number          // Timestamp
}
```

### 2. Threads Table
```typescript
{
  _id: Id<"threads">         // Chat session ID
  userId: Id<"users">        // Who owns this chat
  title: string              // First message (50 chars)
  createdAt: number
  updatedAt: number
}
```

### 3. Messages Table
```typescript
{
  _id: Id<"messages">        // Message ID
  threadId: Id<"threads">    // Which chat
  userId: Id<"users">        // Who sent it
  role: "user" | "assistant"
  content: string            // Message text
  createdAt: number
}
```

## 🔄 How Data Flows

### 1. **User sends message:**
```
User Types → Clicks Send
    ↓
handleSend() triggered
    ↓
await convex.action('chat:chat', {threadId, userId, message})
    ↓
Goes to Convex Backend
```

### 2. **Backend processes:**
```
chat.ts action receives request
    ↓
Save user message: ctx.runMutation("messages:create", {...})
    ↓
Call AI: generateAIResponse(ctx, message, ...)
    ↓
HuggingFace API → Returns response
    ↓
Save AI message: ctx.runMutation("messages:create", {...})
    ↓
Return response to frontend
```

### 3. **Frontend displays:**
```
useQuery('messages:getByThread') listens to database
    ↓
New messages detected
    ↓
queriedMessages updates
    ↓
Component re-renders
    ↓
Messages appear on screen
```

## 🔐 How Authentication Works

### No Traditional Password Authentication

```
1. User enters email + name
2. convex.mutation('users:getOrCreateUser', {email, name})
3. Backend checks if user exists:
   - If YES: Return existing user ID
   - If NO: Create new user
4. Store user ID in localStorage
5. Use for all operations
```

## 📤 How Messages are Stored

### When User Sends Message:

```javascript
// 1. Frontend
const userMessage = "Hello AI"
await convex.action('chat:chat', {
  threadId: "abc123",
  userId: "user456",
  message: userMessage
})

// 2. Backend (chat.ts)
await ctx.runMutation("messages:create", {
  threadId,
  userId,
  role: "user",
  content: message
})

// 3. Database Insert
messages.insert({
  threadId: "abc123",
  userId: "user456",
  role: "user",
  content: "Hello AI",
  createdAt: Date.now()
})

// 4. Frontend Listens
const messages = useQuery(api.messages.getByThread, {threadId})
// Automatically updates when new message inserted
```


## 🔍 Real-time Syncing (Magic of Convex)

### Without Convex:
```
1. Send message to server
2. Wait for response
3. Manually refresh data
4. Update UI
(Slow, manual, error-prone)
```

### With Convex:
```
1. Database updates
2. Convex automatically notifies all subscribed clients
3. useQuery hooks automatically re-fetch
4. Component re-renders
(Automatic, instant, real-time)
```

## 🎯 Key Concepts for Interview

### 1. **What is Convex?**
Backend-as-a-service platform that provides:
- Database + API
- Real-time subscriptions
- Type safety
- No server management

### 2. **How is it different from Firebase?**
| Convex | Firebase |
|--------|----------|
| Queries/Mutations | REST API |
| Type-safe | Runtime validation |
| Full-stack | Backend only |
| Better DX | More mature |

### 3. **What is Real-time Sync?**
When database changes, all connected clients automatically update without manual refresh.

### 4. **How are Indexes Used?**
```typescript
.withIndex("by_threadId", (q) => q.eq("threadId", threadId))
// Finds messages in a specific thread FAST
// Without index: O(n) - scans all messages
// With index: O(log n) - binary search
```

### 5. **What is RAG (Retrieval-Augmented Generation)?**
```
User Question → Search Documents → Add to Context → Send to AI
So AI can answer based on user's documents
```
